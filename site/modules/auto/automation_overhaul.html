>> BACKUP_AUTOMATION

[ PROXMOX_BACKUPS ]

For a long time, the proxmox backups had just been snapshots saved to the local directory in the hypervisor. We want to improve this to include "off-node" bulk storage backups, and fault tolerance.


First, we setup our TrueNAS share for these backups. For the usermapping and groupmapping we're going to be using root:root, this avoids permission errors within proxmox's backup manager. Afterwards, we mount that share, for Proxmox's disks its easiest to set it up in the Proxmox Web UI, where you can add it under the Datacenter storage. For my case, I created prod-backups-01 connecting to 10.50.0.1 (the isolated storage lane detailed in /NET/) /mnt/tank/proxmox_backups with the content filter set to Backup. Instead of keeping all of the backups and eventually flooding the pool, but since we have several terabytes to work with now, rather than a single terabyte, we can be more frugal. I set mine up to:

  • Keep Last: 3
  • Keep Daily: 7
  • Keep Monthly: 2
  • Keep Weekly: 4

This not only gives a decent diversity, but is strikes a good balance between a wide protection window and storage space.


Since I run daily backups, the backups look like this:

  • Days 1-3: Keeps every single backup for the most recent 72 hours.
  • Days 4-7: The Keep Daily 7 rule fills the remaining 4 daily slots.
  • Weeks 2-4: The Keep Weekly 4 rule keeps one anchor backup for each of the last 4 weeks.
  • Months 1-2: The Keep Monthly 2 Rule keeps the final backup of the last two months.

This gives us 10 total backups, with an oldest recovery point of 2 months, since this is not a stagnant server, this is a good tradeoff. If this sat longer without being monitored, checked, or worked on, I would increase the amount, and recovery of the backups.


That was the share, now for the job itself, on Raiden I have configured a backup jobs for all VM/LXCs at 02:00 (AM) to prod-backups-01. I use ZSTD compression, because I do not need these backups to be lossless and ZSTD is a perfect mix of speed, and quality preventing bitrot or dataloss, as it uses a lossless compression algorithm. I also use Snapshot mode so as to keep the services running. I want to work of a 99.99% uptime approach, suspending or stopping for snapshots would be a poor use of the system in a constantly running/serving system.


It is important to note that the snapshots made by Raiden capture the VDisk Data and configuration, this is mostly used for short-term local recovery before changes rather than long-term backups or disaster recovery, more like an "undo".
