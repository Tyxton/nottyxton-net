>> EXHAUSTION_AND_TUNING://INFRASTRUCTURE

Solving an LVM-Thin Swap Explosion and tuning resource management for Proxmox.

>> LVM-THIN SWAP EXPLOSION

[ PROBLEM ]

When I woke up this morning, I went to monitor the servers and noticed that the internet was dragging. I checked into Grafana and noticed that ops-node-01 was completely slammed. When I tried to check into the Web UI for Sonarr, I was met with a 504 error. It had completely locked up. This told me two things: First, ops-node-01 being under heavy load slows down the rest of the network - which is not desired whatsoever; Second, ops-node-01 was completely inaccessible.

[ DIAGNOSIS ]

After realizing this, I went into the Proxmox Web UI to check on the ops-node-01 resource usage directly. I saw that the 1GB Swap was at ~98% usage, while RAM was at ~17%. I then looked at the Raiden host resources and saw two major red flags, IO Delay was sitting at 14.85% and more importantly the Load Average was sitting at 49.60. Raiden only has 12 cores, this means that there are 37 processes just idling, since the CPU usage was only ~9%, this tells me it's hanging on the iowait.

Seeing that the %util was sitting at 99.90% for sda and dm-0 which is an LVM volume, this is CRITICAL. They are under constant load, this is not only going to tear apart the HDD, but it leaves no room for new processes. On top of this, r_await sitting at a peak of 47ms, even for an HDD this is high lag. There is also a lot of random I/O, which is likely caused by this pinned swap. Not only is everything lagging, and overloaded, but the disk itself is thrashing. On a mechanical drive, this will cause permanent damage if not complete failure over time.


The way I have Raiden set up currently, /dev/sda is the boot drive only a 500GB HDD, while /dev/sdb and /dev/sdc are a 1TB mirrored ZFS pool, both HDDs as well. Right now, none of the vdisks are on LVM or LVM-Thin, they are all either on DUAL1T_ZFS or offloaded to Motherbase. This tells me that /dev/sda is being slammed by the container's memory leak. For Proxmox, the LVMs are stored on /dev/sda3 under /data. I decided to check the usual suspects of Proxmox /var/lib/vz, and /var/log with `du -sh /var/lib/vz/*` and `du -sh /var/log` both of those only accounted for ~26.5GB of data. So that isn't the issue. Next, I dediced to check for any files that have been marked deleted, but are being held onto by processes. Since ops-node-01 shows 98% swap, I figured that maybe a massive swap file was set for deletion but being held onto. 

root@raiden:~# lsof +L1
...
Prowlarr  2699555   100000    8u   REG    0,1 2199023255552     0 1539162 /memfd:doublemapper (deleted)
Radarr    2700076   100000    8u   REG    0,1 2199023255552     0 1528711 /memfd:doublemapper (deleted)
...

These two lines were a massive, yet horrifying find. These are "theoretical" - meaning that these could maximally be, not their actual or current size - 2TB files. These are memfd files, meaning that when the physical RAM is exhausted, linux aggressively shoves this data into the swap partition on /dev/sda3. This explains both the 97.57% swap usage and why Raiden is choking. Doublemapper is a known issue in the .NET runtime where especially in certain environments, like mine, where containers with limited memory can glitch, request an excess of virtual memory, then spill out into SWAP.

[ SOLUTION && FURTHER DIAGNOSIS ]

Now that we've tracked down the issue, it's time to fix it.


Immediately, I stopped ops-core-01, as it wouldn't repond to a shutdown. I was initially expecting this to result in releasing the data it held onto, but this wasn't the case. `df -h' showed no change, so I attempted to reinitialize the swap `swapoff -a && swapon -a`, again `df -h` showed minimal/no change.


Though, make one crucial observation, the "missing" 400GB was not in root, it was in /data after all. Likely, the doublemapper ghost files ballooned the host swap volume. The Proxmox UI showed a mismatch in the local-lvm usage, it was only at 0%, but /dev/sda3 still showed 97% full. This tells me that the pool was still reserved. LVM-Thin is an interesting case because it allows for over-provisioning. It reports both the logical usage, and the phsyical allocation. When Prowlarr/Radarr tried to doublemap the 2TB file, it forced the kernel to allocate every physical block available on sda3 for swapspace. >> POST_MORTEM

[ RCA ]

A specific doublemapper/swap bug in the *arr stack caused the container to exceed its assigned RAM, because the LXC had no Swap Limit defined, it began "borrowing" the host's swap space on /dev/sda3. As the swap filled, the HDD was being slammed resulting in high IO wait and locked the CPU scheduler, driving the load average to 49+.

[ SOLUTION ]

Since the LVM-Thin pool would not allow shrinking, despite the overflow, the thin pool was removed to return the physical extents to the volume group. Reassigned the reclaimed space to the root partition and forced a filesystem resize to bring Raiden back to 8% disk usage.

[ PREVENTION ]

Every LXC now has a strict Swap Limit of 512MB. If a process leaks, the container crashes locally instead of taking down the host. Increased ops-node-01 RAM to 4GB to provide enough headroom for .NET JIT operations without hitting the disk.

> ADDITIONAL_ADJUSTMENTS

[ MULTIQUEUE ]

This adjusts how many CPU cores can process network interrupts simultaneously. This prevents a single overloaded core from lagging the other nodes on the network. Generally, you want this to match the provisioned core-count.


Since edge-srv-01 has two cores, we'll set multiqueue to 2. This ensures that different traffic types/protocols aren't fighting for the same interrupt thread.

[ CPU_WEIGHTING ]

We don't want ops-node-01 to hang the front end processes, ops-node-01 should act as a background process, much like it's been set up to do. We'll adjust our critical edge services, edge-srv-01 to 2048. ops-node-01 will be set to 512. This ensures that if both are attempting to use the same CPU core, edge-srv-01 gets 4x the priority. This is necessary as it hosts all of the public facing services that we want to always work, and always work fast. We do this with `[qm/pct] set [vm/lxc id] --cpuunits 2048`

[ SWAPPINESS ]

Since Raiden's default swappiness is set to 60, we want to lower this so that it only uses swap when necessary, we want it to use the RAM as much as possible. 

sysctl -w vm.swappiness=10 && \
echo "vm.swappiness=10" >> /etc/sysctl.conf

Now with these final adjustments made, our services will be more stable, and more reactive to the environment. This is much better for a production server.
